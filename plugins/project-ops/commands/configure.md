Run an interactive setup flow to configure the project-ops plugin for the current repository.

## Step 1: Detect org/repo

Parse the git remote to detect the GitHub org and repo name:

```bash
git remote get-url origin
```

Extract org and repo from the SSH (`git@github.com:org/repo.git`) or HTTPS (`https://github.com/org/repo.git`) URL. Confirm the detected values with the user.

## Step 2: List available projects

Query the organization's GitHub Project V2 boards:

```bash
gh api graphql -f query='
{
  organization(login: "<ORG>") {
    projectsV2(first: 20) {
      nodes {
        number
        title
        url
        closed
      }
    }
  }
}' --jq '.data.organization.projectsV2.nodes[] | select(.closed == false) | "#\(.number)\t\(.title)\t\(.url)"'
```

Present the list of open projects to the user.

## Step 3: Ask which project to use

Ask the user to select a project number from the list. Also offer the option to create a new project by running `/project-ops:bootstrap`.

## Step 4: Fetch all project metadata

Run this query to get the project ID, every custom field with its option IDs, and all sprint iterations (active and completed):

```bash
gh api graphql -f query='
{
  organization(login: "<ORG>") {
    projectV2(number: <NUMBER>) {
      id
      fields(first: 30) {
        nodes {
          ... on ProjectV2Field {
            id name dataType
          }
          ... on ProjectV2SingleSelectField {
            id name
            options { id name }
          }
          ... on ProjectV2IterationField {
            id name
            configuration {
              iterations { id title startDate duration }
              completedIterations { id title startDate duration }
            }
          }
        }
      }
    }
  }
}'
```

## Step 5: Write config

Determine the config file location: `$CLAUDE_PROJECT_DIR/.claude/project-ops.yaml` if `CLAUDE_PROJECT_DIR` is set, otherwise `<git-root>/.claude/project-ops.yaml`.

Write the full YAML file using all data fetched in Step 4. The format must match exactly what `project-config.sh` expects — use yq-compatible YAML (no anchors, no aliases).

For every field returned by the query:
- Single-select fields → include `id` + `options` array (each with `id` and `name`)
- Iteration fields → include `id` + `iterations` array (active) + `completed_iterations` array
- Plain fields (date, text, etc.) → include `id` only

Use the field's exact name as returned by GitHub (including spaces and special characters) as the YAML key under `fields:`. Quote keys that contain special characters.

Example structure:

```yaml
---
org: <org>
repo: <repo>
project_number: <number>
project_id: <PVT_...>
fields:
  Status:
    id: PVTSSF_...
    options:
      - id: "f75ad846"
        name: "Todo"
      - id: "47fc9ee4"
        name: "In Progress"
      - id: "98236657"
        name: "Done"
  Priority:
    id: PVTSSF_...
    options:
      - id: "21061acd"
        name: "P0 - Critical Path"
      - id: "45902923"
        name: "P1 - High"
      - id: "64382e2e"
        name: "P2 - Medium"
      - id: "8ed87257"
        name: "P3 - Low"
      - id: "1b2e3329"
        name: "P4 - Future"
  "Critical Path":
    id: PVTSSF_...
    options:
      - id: "..."
        name: "..."
  Sprint:
    id: PVTIF_...
    iterations:
      - id: "..."
        title: "Sprint 1: ..."
        startDate: "2026-02-17"
        duration: 14
    completed_iterations:
      - id: "..."
        title: "Sprint 0"
        startDate: "2026-02-03"
        duration: 14
  "Start Date":
    id: PVTF_...
  "Target Date":
    id: PVTF_...
  "Design Doc":
    id: PVTF_...
  "Plan Doc":
    id: PVTF_...
---

# Project Ops Configuration

This file was generated by `/project-ops:configure`. It links this repository to
GitHub Project V2 board #<number> and caches all field and option IDs locally so
no API calls are needed at runtime.

To refresh (e.g. after adding a new sprint or field), run `/project-ops:configure` again.
```

Write all fields returned by the query — do not skip any. Include every single-select option and every iteration (both active and completed).

## Step 6: Confirm

Tell the user that setup is complete and summarize what was written (project ID, fields found, sprint count). Suggest they run `/project-ops:status` to verify everything is working.
